# プロジェクトルール

## 1. 開発フロー

### コード追加・変更は設計書を先に書く

- 実装前に `設計書.md` に変更内容・理由を追記してから実装を始める
- 「動かしながら考える」はしない

### コミット・プッシュは main ブランチで管理

- ブランチは作らず、すべて `main` ブランチで作業する
- お試しアプリのため、シンプルな運用を優先する

---

## 2. セキュリティ

### APIキーの管理

- `.env` ファイルは **絶対にコミットしない**（.gitignore で除外済み）
- API キーをコードに直接書かない（ハードコードしない）
- `.env.example` にはダミー値のみ記載する

### 入力値の検証

- ユーザーの入力は必ずサーバー側でもバリデーションする
- フロント側だけのバリデーションに頼らない

---

## 3. コスト管理

- 新しい機能を追加するときは、トークン使用量への影響を設計書に明記する
- `max_tokens` / レートリミット / 履歴件数の設定は変更前に設計書でレビューする

---

## 4. コミットルール

### コミットメッセージのフォーマット

```
<種別>: <変更内容を一言で>
```

| 種別       | 使うタイミング                     |
| ---------- | ---------------------------------- |
| `feat`     | 新機能の追加                       |
| `fix`      | バグ修正                           |
| `docs`     | 設計書・手順書・README の更新      |
| `style`    | デザイン・見た目の変更             |
| `refactor` | 動作を変えないコードの整理         |
| `chore`    | 設定ファイルや依存パッケージの更新 |

**例:**

```
feat: レートリミット機能を追加
fix: 会話履歴が消えないバグを修正
docs: 設計書にコスト方針を追記
```

### コミット前チェック

- [ ] `docker compose up` で正常に動作することを確認した
- [ ] `.env` をステージングに含めていない（`git status` で確認）
- [ ] 設計書の更新が必要な変更の場合、設計書も一緒にコミットした

---

## 5. ファイル管理

| ファイル       | 用途                         | コミット  |
| -------------- | ---------------------------- | --------- |
| `設計書.md`    | 機能・構成の設計             | ✅ する   |
| `README.md`    | 起動手順・概要               | ✅ する   |
| `手順書_*.md`  | 非エンジニア向け作り方ガイド | ✅ する   |
| `ルール.md`    | このファイル                 | ✅ する   |
| `.env`         | APIキー（秘密情報）          | ❌ しない |
| `.env.example` | 環境変数のテンプレート       | ✅ する   |

---

## 6. その他（推奨）

- **動作確認してからコミットする**（壊れた状態で push しない）
- **設計書・手順書もコードと同じタイミングで更新する**（ドキュメントが古くならないように）
- **AIが生成したコードも必ず自分で読む**（内容を把握してからコミットする）
